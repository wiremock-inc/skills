# Making stubs data-driven

If asked to convert a collection of stubs to be data-driven i.e. to use data from a data source,
follow these guidelines:

1. Start by defining a data source derived from the structure and data in the stubs. Create this in WireMock Cloud using the `create_data_source` tool.
2. Update each stub in the collection indicated by the user using the `update_stub_mapping` tool to use the data source. This involves:
    1. Adding a `data-source` request matcher to the stub that matches on the data source ID.
    2. Enabling response templating on the stub.
    3. Updating the response body to use Handlebars expressions to retrieve data from the data source.
3. If instructed by the user, generate any additional stubs required, also utilising the datasource, and import them using `import_stubs_to_mock_api`.
4. Look up documentation article `validating_and_fixing`, validate the stubs and if necessary apply fixes.

The response generated by the Handlebars template must adhere to the same structure as the original response in the original version of the stub - all fields must be present.

If pagination is evident from the query parameters and response body metadata, ensure to enable pagination as described later in this article.

## Example - converting a simple, non-paginated list stub to be data-driven

Suppose we have the following stub response body, which lists GitHub users:

```json
[
  {
    "login": "mojombo",
    "id": 1,
    "node_id": "MDQ6VXNlcjE=",
    "avatar_url": "https://avatars.githubusercontent.com/u/1?v=4",
    "gravatar_id": "",
    "url": "https://api.github.com/users/mojombo",
    "html_url": "https://github.com/mojombo",
    "followers_url": "https://api.github.com/users/mojombo/followers",
    "following_url": "https://api.github.com/users/mojombo/following{/other_user}",
    "gists_url": "https://api.github.com/users/mojombo/gists{/gist_id}",
    "starred_url": "https://api.github.com/users/mojombo/starred{/owner}{/repo}",
    "subscriptions_url": "https://api.github.com/users/mojombo/subscriptions",
    "organizations_url": "https://api.github.com/users/mojombo/orgs",
    "repos_url": "https://api.github.com/users/mojombo/repos",
    "events_url": "https://api.github.com/users/mojombo/events{/privacy}",
    "received_events_url": "https://api.github.com/users/mojombo/received_events",
    "type": "User",
    "user_view_type": "public",
    "site_admin": false
  },
  {
    "login": "defunkt",
    "id": 2,
    "node_id": "MDQ6VXNlcjI=",
    "avatar_url": "https://avatars.githubusercontent.com/u/2?v=4",
    "gravatar_id": "",
    "url": "https://api.github.com/users/defunkt",
    "html_url": "https://github.com/defunkt",
    "followers_url": "https://api.github.com/users/defunkt/followers",
    "following_url": "https://api.github.com/users/defunkt/following{/other_user}",
    "gists_url": "https://api.github.com/users/defunkt/gists{/gist_id}",
    "starred_url": "https://api.github.com/users/defunkt/starred{/owner}{/repo}",
    "subscriptions_url": "https://api.github.com/users/defunkt/subscriptions",
    "organizations_url": "https://api.github.com/users/defunkt/orgs",
    "repos_url": "https://api.github.com/users/defunkt/repos",
    "events_url": "https://api.github.com/users/defunkt/events{/privacy}",
    "received_events_url": "https://api.github.com/users/defunkt/received_events",
    "type": "User",
    "user_view_type": "public",
    "site_admin": false
  }
]
```

We want to convert this to be data-driven, using a CSV data source with the following columns:

| Column Name | Type |
|-------------|------|
| login | STRING |
| id | INTEGER |
| node_id | STRING |
| avatar_url | STRING |
| gravatar_id | STRING |
| url | STRING |

Then the stub could need a custom matcher of this form to be added:
```json
{
  "customMatcher" : {
    "name" : "custom-and-matcher",
    "parameters" : {
      "matchers" : [ {
        "name" : "data-exists",
        "parameters" : {
          "id" : "nw4yk",
          "matcherEnabled" : false,
          "query" : ""
        }
      } ]
    }
  }
}
```
`id` is the ID of the datasource we created in the previous step.
`matcherEnabled` is false as we want to match the stub even if there are 0 rows returned. We would
set this to `true` if we wanted to only match if there is at least one row returned e.g. for a single record lookup.
`query` is a SQL WHERE clause which can be empty if we want to return all rows in the data source, but can be used
to limit the set in more advanced cases such as pagination and filtering.

Then the stub response body would be updated to use Handlebars expressions to retrieve data from the data source:

```handlebars
{{#formatJson}}
[
  {{~#arrayJoin ',' data.items as |user|~}}
  {
    "login": "{{user.login}}",
    "id": {{user.id}},
    "node_id": "{{user.node_id}}",
    "avatar_url": "{{user.avatar_url}}",
    "gravatar_id": "{{user.gravatar_id}}",
    "url": "{{user.url}}"
  }
  {{/arrayJoin}}
]
{{/formatJson}}
```

## Single record lookup stubs

If the stub is for a single record lookup, the SQL query in the custom matcher should be set to match on the ID in the URL.
For example, assuming an endpoint with path template `/users/{userId}`, the custom matcher would be:

```json
{
  "customMatcher" : {
    "name" : "custom-and-matcher",
    "parameters" : {
      "matchers" : [ {
        "name" : "data-exists",
        "parameters" : {
          "id" : "nw4yk",
          "matcherEnabled" : true,
          "query" : "id = '{{request.path.userId}}'"
        }
      } ]
    }
  }
}
```

## Pagination
Pagination is typically implemented in one of three ways:
1. Page/limit
2. Offset/limit
3. Cursor

Pagination is usually controlled via query parameters.
For example, a page/limit implementation might use `?page=1&limit=10` to retrieve the first 10 records.

Response bodies typically contain pagination metadata of two kinds:
1. Metadata - usually some combination of: page number, total pages, total items, limit.
2. Links - usually some combination of: first, previous, next, last.

### Example - page/limit pagination
Assuming an API that takes `page` and `limit` query parameters, we would add a query attribute to the custom matcher
that calculates the offset parameter the builds an appropriate where clause. It also sets defaults for `page` and `limit` if they are not provided:

```json
{
  "customMatcher" : {
    "name" : "custom-and-matcher",
    "parameters" : {
      "matchers" : [ {
        "name" : "data-exists",
        "parameters" : {
          "id" : "nw4yk",
          "matcherEnabled" : true,
          "query" : "{{val request.query.limit.0 or=10 assign='limit'}}{{val request.query.page.0 or=1 assign='page'}}{{val (math (math page '-' 1) '*' limit) assign='offset'}} LIMIT {{limit}} OFFSET {{offset}}"
        }
      } ]
    }
  }
}
```

The response body would then need to include pagination metadata and links e.g.

```handlebars
{{val request.query.page or='1' assign='page'}}
{{#assign 'limit'}}{{#if request.query.limit}}&limit={{request.query.limit}}{{/if}}{{/assign}}
{
  "metadata": {
    "page": {{page}},
    "limit": {{limit}}
  },
  "links": {
    "self": "{{request.baseUrl}}{{request.url}}",
    "next": "{{request.baseUrl}}/users?page={{math page '+' 1}}{{limit}}"{{#and (neq page 1) (neq page '1')}}{{#and (neq page 1) (neq page '1')}},
    "prev": "{{request.baseUrl}}/users?page={{math page '-' 1}}{{limit}}"{{/and}}
  }
}
```
